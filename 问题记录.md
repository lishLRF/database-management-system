## AI误导探索表强于修改

### 现象

**用户输入**: "在guide_rail表插入型号GR-2024的导轨数据"

**期望行为**: Planner 生成 2 个任务 — ① explore 任务（探索表结构和现有数据）② insert 任务（生成 INSERT SQL）

**实际行为**: Planner 只生成 1 个任务 — `{"type":"explore", "target":"guide_rail", "desc":"探索guide_rail表结构和现有数据", "deps":""}`

**后果**:

- Executor 执行 explore 任务时调用了 7 次工具（describe_table、query_data 等）
- 但 explore 任务的目标是"探索"，executor prompt 没有指示"探索完后输出 INSERT SQL"
- 最终 `taskResults` 为空（0个SQL输出），Agent 抛出异常 `RuntimeException: Agent failed to generate any SQL`

---

### 原因分析

**根本原因**: Planner prompt 的 INSERT 规则表述不够强制

**原 Planner prompt (INSERT部分)**:

```
case "insert":
    extra.append("INSERT mode. MUST plan explore task FIRST, then insert task. Explore=describe_table+query_data.\n");
```

**AI 理解偏差**:

1. "MUST plan explore task FIRST" → AI 理解成"必须先规划 explore"，**但没理解成"还要规划 insert"**
2. 用户说"插入数据"，AI 认为"先探索表"是合理的第一步，**就停在这一步了**
3. LLM 倾向于保守规划（fewer tasks = simpler plan），如果规则模糊就会少规划任务

**类比**: 就像对小孩说"你必须先洗手"，小孩洗完手就不吃饭了——因为你没说"洗手之后还要吃饭"。

---

### 解决方案对比

#### 方案1: 强化 Planner prompt 规则 ✅（最终选择）

**改动**:

```java
case "insert":
    extra.append("INSERT mode. MANDATORY: plan TWO tasks — (1) explore task with type='explore' to describe_table+query_data, (2) insert task with type='insert' that depends on explore task.\n");
```

**优点**:

- ✅ 明确数量要求（TWO tasks）
- ✅ 明确类型要求（type='explore' 和 type='insert'）
- ✅ 明确依赖关系（insert depends on explore）
- ✅ 改动小（仅修改 Planner prompt）
- ✅ 通用性强（UPDATE/DELETE 同理）

**缺点**:

- ⚠️ 依然依赖 LLM 理解，无法 100% 保证（但风险已降到极低）

---

#### 方案2: Planner 后处理补任务

**思路**: 如果检测到 Planner 只生成了 explore 任务，代码自动补一个 insert 任务

```java
// Pseudo-code
if (intentType.equals("insert") && tasks.size() == 1 && tasks.get(0).get("type").equals("explore")) {
    tasks.add(Map.of("type", "insert", "target", tableName, "desc", "插入数据到" + tableName, "deps", tasks.get(0).get("desc")));
}
```

**优点**:

- ✅ 100% 保证有 2 个任务（代码强制）
- ✅ 兜底方案，防止 Planner 失败

**缺点**:

- ❌ 硬编码逻辑，不优雅（后处理补丁 vs 让 AI 正确规划）
- ❌ 补的任务描述可能不准确（"插入数据到guide_rail" 太泛，不如 Planner 生成的"插入型号GR-2024的导轨"具体）
- ❌ 维护成本高（每个 intent 都要写一套补任务逻辑）

---

#### 方案3: 改用 structured output 强制 JSON schema

**思路**: 调用 LLM 时用 `response_format: {type: "json_schema", schema: {...}}` 强制返回符合结构的 JSON

```json
{
  "type": "object",
  "properties": {
    "tasks": {
      "type": "array",
      "minItems": 2,  // 强制至少2个任务
      "items": {
        "type": "object",
        "properties": {
          "type": {"enum": ["explore", "insert", "update", "delete", "select"]},
          "target": {"type": "string"},
          "desc": {"type": "string"},
          "deps": {"type": "string"}
        }
      }
    }
  }
}
```

**优点**:

- ✅ JSON 结构保证（不会返回非法格式）
- ✅ `minItems: 2` 强制至少 2 个任务

**缺点**:

- ❌ DeepSeek API 可能不支持完整的 OpenAI structured output（需测试）
- ❌ `minItems: 2` 对 SELECT 场景不适用（查询可能只需 1 个任务）
- ❌ 需要按 intent 动态调整 schema（INSERT 要 2 个，SELECT 要 1 个），复杂度高

---

#### 方案4: 引入 "写前必探索" 的全局规则

**思路**: 在 executor prompt 中加入"如果是 write 操作（INSERT/UPDATE/DELETE），必须先调用 describe_table 和 query_data"

**优点**:

- ✅ 即使 Planner 失败，Executor 也能自救
- ✅ 防御性编程

**缺点**:

- ❌ 与当前"Planner 规划 explore 任务"的架构冲突（职责混乱：到底谁负责探索？）
- ❌ 无法解决"没有 insert 任务"的问题（Executor 探索完了，但没有后续任务告诉它"生成 INSERT"）

---

### 最终选择：方案1（强化 Planner prompt）

**选择理由**:

1. **符合架构原则** — "Planner 负责规划，Executor 负责执行"，不应该用后处理补丁（方案2）或 Executor 兜底（方案4）破坏职责边界
2. **成本最低** — 只改一段文字，无需新增代码逻辑或调整 API 调用方式
3. **效果可验证** — 通过日志 `[Agent] Tasks=2` 立即看到是否生效，问题清晰
4. **可扩展性好** — UPDATE/DELETE 同理应用（已同步修改为 MANDATORY TWO tasks）
5. **符合 LLM 使用最佳实践** — 明确的、数量化的指令（"plan TWO tasks"）比模糊的指令（"MUST plan explore task FIRST"）更有效

**风险评估**:

- ⚠️ LLM 仍有 ~1% 概率误解（无法 100% 保证）
- ✅ 但方案2可作为兜底（如果方案1验证失败，5分钟加上后处理逻辑）

**验证计划**:

1. 重启后端 `mvn spring-boot:run`
2. 触发 INSERT 请求："在guide_rail表插入型号GR-2024"
3. 检查日志 `[Agent] Tasks=` 是否为 2
4. 如果为 2 → 方案1成功 ✅
5. 如果还是 1 → 启用方案2（后处理补任务）作为兜底

## 分块策略Agent流程不稳定

### 现象

**触发操作**: 上传 Markdown 文档 → 点击 2️⃣ 分块 → AI 分析

**期望行为**: AI 返回完整的分块策略 JSON 数组，每块包含 `{start, end, title, type, reason}`，前端展示策略列表 + 右侧预览切片内容

**实际行为**: 出现以下多种不稳定表现：

| 表现                     | 后端错误                                                                                                           | 频次 |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------ | ---- |
| 分块返回 0 块，静默失败  | `[MarkdownAgent] Parse error: No content to map due to end-of-input at [Source: (String)""; line: 1, column: 0]` | 高频 |
| 分块返回 0 块，JSON 截断 | `[MarkdownAgent] Parse error: Unexpected end-of-input in field name` 或 `Unterminated string`                  | 中频 |
| 分块成功但策略质量差     | 无报错，但 chunk 边界不合理（切在段落中间、覆盖不全）                                                              | 低频 |
| 分块数量波动大           | 同一文档两次分块结果差异大（第一次 3 块，第二次 8 块）                                                             | 中频 |

**后果**:

- 用户点击分块后，进度条跑完但策略列表为空，右侧预览无内容
- 无法进入后续"确认存库 → 导出"流程
- 重新点击分块结果可能不同，行为不可预期

---

### 原因分析

#### 根因 1: maxTokens 硬编码 4096，忽略用户配置 🔴 主因

**位置**: `MarkdownAgent.java:103-104`（2026-06-14 已修复）

**原代码**:

```java
String response = callAI(config, prompt,
    "Return JSON array of chunks.", 0.2, 4096);  // ← 硬编码 4096
```

**机制**: `callAI()` 的第5个参数直接传给 DeepSeek API 的 `max_tokens` 字段。4096 tokens 对长文档的 JSON 输出严重不足：

```
典型长文档（5000字中文）分块输出量估算：
- 每个 chunk JSON 约 80-150 tokens
- 10 个 chunk ≈ 800-1500 tokens  ← 看起来够？
- 但 AI 会额外输出解释性文字、markdown 包裹、reason 字段冗长
- 实际输出常超过 2000-4000 tokens
- 加上 AI 可能输出多轮分析文本后再输出 JSON
- 4096 经常被撑爆 → 输出截断 → JSON 不完整
```

**为什么有时成功有时失败**: 取决于 AI 当次的"话多程度"。如果 AI 在 JSON 前输出分析文字，token 消耗更大；如果直接输出 JSON，可能刚好卡在 4096 以内。

**修复** (2026-06-14):

```java
int chunkMaxTokens = Math.max(config.getMaxTokens() != null ? config.getMaxTokens() : 4096, 16384);
String response = callAI(config, prompt, "...", 0.2, chunkMaxTokens);
```

- 读取用户 AI 配置中的 maxTokens
- 保底 16384 tokens（即使未配置也够用）
- 用户可设到 100000+ 获得更大余量

---

#### 根因 2: 一次性全量输出架构 — 无工具调用拆分 🟡 架构问题

**对比 SQL Agent**:

| 维度       | SQL Agent                                                    | Markdown Agent                             |
| ---------- | ------------------------------------------------------------ | ------------------------------------------ |
| 任务粒度   | 多步工具调用（describe_table → query_data → generate SQL） | 单次 AI 调用输出完整 JSON                  |
| 输出大小   | 每步几百 tokens                                              | 一次性几千 tokens                          |
| 出错恢复   | 单步失败可重试，不影响其他步                                 | 一个 JSON 字段错误 → 整个解析失败         |
| 上下文控制 | 每步注入所需信息，逐步累积                                   | 一次性注入全文 + schema + 历史，上下文臃肿 |

**SQL Agent 稳定性的关键**: 函数调用协议 `[TOOL:name{params}]` 把复杂任务拆成**可控的小步**。每步 AI 只需要输出一个工具调用（几十 tokens），而不是完整答案（几千 tokens）。

**Markdown Agent 的架构缺陷**: `proposeChunks()` 将"理解全文 → 识别语义边界 → 分类 → 输出完整 JSON 数组"全部压在一次 `callAI()` 中。这是典型的 **"一次性全量输出"反模式**。

---

#### 根因 3: 多轮迭代是假象 — mergeChunks 实为替换 🟡 设计问题

**位置**: `MarkdownAgent.java:177-178`

```java
private List<Map<String, Object>> mergeChunks(
    List<Map<String, Object>> existing, List<Map<String, Object>> updated) {
    return updated.isEmpty() ? existing : updated;  // ← 直接用 updated 替换
}
```

**设计意图**: 多轮迭代（最多4轮），每轮在前一轮基础上精炼分块策略。

**实际效果**: `mergeChunks()` 在新结果非空时**直接丢弃旧结果**。这意味着：

- 第1轮 AI 输出5块 → chunks = [5块]
- 第2轮 AI 输出8块 → chunks = [8块]（第1轮的5块被丢弃！）
- 第3轮 AI 返回空 → chunks = [8块]（保持不变），break

本应"每轮增量优化"的迭代，退化成了"每轮重新覆盖"的重试。如果某一轮 AI 质量差（切分过于粗糙），上一轮更好的结果会被覆盖。

---

#### 根因 4: 无验证环节 — 缺少覆盖完整性检查 🟡 缺失能力

SQL Agent 有 Validate 阶段：执行生成的 SQL，检查是否成功、结果是否正确。

Markdown Agent 的 `parseChunkResponse()` 只做 JSON 解析：

```java
} catch (Exception e) {
    System.err.println("[MarkdownAgent] Parse error: " + e.getMessage());
    return List.of();  // 静默返回空列表
}
```

**缺少的检查**:

- ❌ chunk 的 start/end 是否连续覆盖全文？（有没有漏掉的段落？）
- ❌ chunk 的 start/end 是否有重叠？（同一段内容被分到两个 chunk？）
- ❌ chunk 的 type 分类是否合理？（全部分成 structured_data？）
- ❌ chunk 的边界是否落在合理位置？（段落中间 vs 段落边界？）

---

#### 根因 5: 提示词为英文，中文文档匹配度差 🟢 次要

`buildChunkPrompt()` 的6条规则是英文硬编码。中文 Markdown 文档（技术规格书、设计文档）的语义结构与英文不同（标题层级、列表格式、表格密度），英文规则可能导致 AI 对中文文档的语义边界判断不准确。

---

### 解决方案

架构重构为 **四阶段管线**（2026-06-14 全部实施完成）：

```
Phase 0a: 代码预扫       (0 AI)
Phase 0b: AI验证候选标题  (1次批量AI调用)
Phase 1:  原子拆分       (代码切分 + AI处理超长段)
Phase 1.5: 纯度预检      (0 AI 计算 + 1次批量AI确认混合段)
Phase 2:  工具调用分类   (N次AI调用，每次~100-500 tokens)
Phase 3:  相邻合并+验证  (0 AI，纯代码)
```

---

#### 子问题1: maxTokens 硬编码 → AI返回空/截断 🔴 已修复

**根因**: 所有 `callAI()` 调用点硬编码 `max_tokens=4096`，但实际输出远大于此。DeepSeek V4 Pro 是推理模型，token 配额需同时分配给 `reasoning_content` 和 `content`，4096 连推理都不够。

**修复**:

- 新增 `outputTokens(config)` 辅助方法，统一输出上限 = `max(config.maxTokens / 3, 16384)`
- 新增 `toolLoopTokens(config)` 辅助方法 = `min(outputTokens, 16384)`，用于工具调用循环（防止推理模型 over-reasoning 淹掉工具调用）
- `callAI()` 新增 reasoning model 兼容：当 `content` 为空时自动从 `reasoning_content` 提取工具调用或兜底文本
- 覆盖文件：`MarkdownAgent.java`、`AgentService.java`、`AiService.java`

---

#### 子问题2: 一次性全量输出 → JSON截断/解析失败 🔴 已修复

**根因**: 旧架构 `proposeChunks()` 将"理解全文 → 识别语义边界 → 分类 → 输出完整 JSON 数组"全部压在一次 `callAI()` 中。

**修复 — 函数调用化（对标 SQL Agent）**:

- Phase 2 实现 `[TOOL:classify_segment{...}]` 工具调用循环
- 每次 AI 只需输出一个工具调用（~100-500 tokens），永不截断
- 代码实时验证工具调用 → 即时反馈给 AI
- 最大 80 轮迭代，支持持续修正

---

#### 子问题3: 多轮迭代是假象 → 结果不稳定 🔴 已修复

**根因**: 旧版 `mergeChunks()` 直接 `return updated`，多轮结果互相覆盖而非增量精炼。

**修复**: 废除多轮循环设计。新架构用 **Phase 2 工具调用循环** 替代：

- AI 逐个片段分类，每步状态持久化
- 单个片段失败不影响其他片段
- Phase 3 合并相邻同类型 chunk 实现最终聚合

---

#### 子问题4: 标题检测太窄 → 长文档只切出 1-2 个段落 🟡 已修复

**根因**: Phase 0 只认 `#`/`##` markdown 标题，中文编号（一、1.1）和粗体字标题被忽略。

**修复 — Phase 0a+0b 双阶段标题识别**:

| 类型                  | 来源                                                  | 处理方式                            |
| --------------------- | ----------------------------------------------------- | ----------------------------------- |
| `hardHeadings`      | `#`/`##` markdown 标题                            | 无条件信任                          |
| `candidateHeadings` | 中文编号 `一、`/`1.1`/`（一）` + `**粗体行**` | AI 批量验证（含上下文 80+120 字符） |

AI 验证对每个候选提供上文 + 候选行 + 下文，判断是否为断义标题。粗体字如果只是强调（如行内 `**注意**`）会被 AI 拒绝。

---

#### 子问题5: 段落边界无法拆分数混合段 → 结构化数据与语义文字混在一起 🟡 已修复

**根因**: Phase 1 依赖结构边界（标题、`\n\n`、表格）切分。当一段文本"前半是数据、后半是语义"但中间没有 `\n\n`，会被当作一个原子段送入 Phase 2，AI 只能整个分类。

**修复 — 方案2+方案3 组合（纯度预检 + split_segment 工具）**:

**Phase 1.5 `purityPreCheck()`**（代码计算 + 1 次 AI 批量确认）:

```
数据密度 = 数据特征字符数(数字+单位+键值对+表格) / 总有效字符数

< 15%  → 纯语义   → 直接 classify_segment
15%-75% → 疑似混合 → AI 批量确认哪些需要 split
> 75%  → 纯数据   → 直接 classify_segment 为 structured_data

混合检测范围保留 15%-75% 的弹性区间，
避免边界值（14% vs 16%）导致的分类跳变。
```

**Phase 2 新增 `[TOOL:split_segment{...}]` 工具**:

```
[TOOL:split_segment{"start":0,"end":320,"split_at":185,
  "left_type":"structured_data","left_title":"电机参数",
  "right_type":"semantic_content","right_title":"裕度分析"}]
```

- 只能对 `confirmedMixed` 集合中的段使用
- 代码验证 split_at 在 (start, end) 范围内
- 拆开后直接创建两个 chunk 加入 results

---

#### 子问题6: 无验证环节 → 分块覆盖不全/重叠 🔴 已修复

**根因**: 旧版无任何验证，AI 输出的 chunk 可能有 gap（未覆盖的段落）或 overlap（重复覆盖）。

**修复 — Phase 3 五维验证（纯代码）**:

| 检查项        | 说明                                    | 策略                                    |
| ------------- | --------------------------------------- | --------------------------------------- |
| D1 覆盖完整性 | 所有 chunk 必须连续覆盖 `[0, docLen]` | 发现 gap → 自动补"未分类"块            |
| D2 重叠检测   | 相邻 chunk 不能有重叠区间               | 自动调整边界至中点                      |
| D3 边界吸附   | chunk 边界应在 `\n\n` 处 ±50 字符    | 自动吸附到最近段落边界                  |
| D5 过小检测   | chunk 内容 < 20 字符                    | UI 高亮警告（不强制合并，保留用户决策） |

## 复杂类型（JSONB/HSTORE/ARRAY）在查询结果中显示为 `[object Object]`

### 现象

**触发场景**: PostgreSQL 表中有 `jsonb` 类型的列，执行 `SELECT * FROM table` 后在前端表格中显示 `[object Object]`，无法看到 JSON 的实际内容。

**错误日志** (行编辑模式):

```
错误: 类型json的输入语法无效 详细：Token "object" is invalid. 位置：443
```

**连锁影响**:

- 查询结果表格无法查看/编辑 JSON 列
- 行编辑模式下，JSON 列显示为 `[object Object]`，点击更新时报 JSON 语法错误
- 智能填表的 UPDATE 生成可能包含 `SET "col" = [object Object]` 这样的无效 SQL

### 原因

`SqlExecutionService.formatResult()` 将 `ResultSet` 的值转为字符串时直接调用 `.toString()`:

```java
// 伪代码
Object value = rs.getObject(columnIndex);
row.put(columnName, value != null ? value.toString() : null);
```

对于 `jsonb` 类型:

- PostgreSQL JDBC driver 返回的是 `org.postgresql.util.PGobject` 实例
- `PGobject.toString()` 返回 `"Object"` 或其他字符串，不是实际的 JSON 值
- 正确的做法是调用 `PGobject.getValue()` 获取原始 JSON 字符串

对于 `hstore`:

- 返回 `java.util.Map`，`toString()` 输出 `{key=value}` 格式

对于数组类型:

- 返回 `java.sql.Array`，需要 `Array.getArray()` 后再序列化

### 标签

`[PENDING]` — 遗留问题，等待 Task #17 (JSON/YAML/XML 复杂类型解析) 功能完成后统一解决。

### 已确认的影响范围

| 端点                                               | 影响                                                      |
| -------------------------------------------------- | --------------------------------------------------------- |
| `POST /api/sql/execute` (SELECT)                 | 结果表格显示 `[object Object]`                          |
| `POST /api/sql/table/{table}/rows` (行搜索)      | 行编辑模式加载数据时 JSON 列显示异常                      |
| `POST /api/sql/table/{table}/rows/{pk}` (取单行) | 同上                                                      |
| AI补全空值                                         | AI 可能收到 `[object Object]` 作为列值，生成无效 UPDATE |

### 计划修复方案（草案）

1. **后端** `SqlExecutionService.formatResult()` 增加类型检测:

   - `PGobject` → 调用 `getValue()`，对 JSON 类型执行 `JSON.parse` 验证后返回
   - `java.sql.Array` → 转 Java 数组再 JSON 序列化
   - `Map` → JSON 序列化
2. **前端** `ResultTable.vue` 按列类型决定渲染方式:

   - 普通类型 → 文本
   - JSON → 点击可展开的格式化查看器
   - XML → 同上
   - YAML → 同上
3. **前端** 行编辑表单中 JSON 列值的显示:

   - 显示格式化 JSON 预览（截断到 200 字符）
   - 点击可弹窗编辑完整 JSON
